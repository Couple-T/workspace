A configurable AI product team, in a repo

A reusable template that turns a folder into a working software team: a flat crew of Claude agents that takes a ticket from idea to shipped code — across every repo it touches — while staying portable across your code host, tracker, and tooling.

01What is aiworks?

aiworks is the workspace-template: a ready-made setup where a team of AI agents — from a CEO down to developers, QA, and reviewers — picks up a ticket and carries it all the way to merged, shipped code.

It runs across many repos at once (app, backend, end-to-end tests, …) and coordinates them in the right order. Everything that ties the team to one company — your code host, your issue tracker, your quality gate — is swappable. You describe your world once in a single config file, and the agents, skills, and workflows all read from it.

03From idea to shipped: three workflows

Work moves through three workflows you can run on their own or back-to-back. Each one hands a clean artifact to the next, so a fuzzy idea becomes designs, then a backlog, then shipped code.

Business Requirements. Turns a roadmap phase or a free-text directive into a BRD, written to the repo (and optionally your doc space).

  1. Research — market, competitors, pricing, opportunity.
  2. Strategy — CEO sets vision, goals, success metrics, scope.
  3. Product — CPO defines the feature set + unit economics.
  4. Feasibility — CTO weighs risk, approach, cross-repo + ADR flags.
  5. Write — Documentor assembles & publishes the BRD.

Product + design + ticketing. Turns an approved BRD into production-ready designs and a ready-for-dev backlog.

  1. Intake — CPO turns the BRD into prioritized feature briefs (flagged UI-bearing or not).
  2. Design — conditional: for UI features, UX planner → graphic assets → Figma frames (in parallel). Skipped entirely for non-UI work.
  3. Ticketing — Product Owner writes one self-contained ticket per feature, linking its Figma frame.

Build & ship one ticket. Takes a single ticket end-to-end across every repo it touches. Detailed in the next section.

  1. Scope → Kickoff → Build → Open PR → Review — per repo, in dependency waves.
  2. Merge → Integration → Distribute → Summary — cross-repo join and ship.

04Inside the dev-cycle

The dev-cycle is the engine. It takes one ticket and runs every repo that ticket touches through the same nine phases — building repos in dependency waves (a backend before the app, the app before its end-to-end tests) and merging them upstream → downstream. When a ticket touches just one repo, it quietly collapses to a simple single-repo flow.

#PhaseWhat happens
1ScopeCTO classifies which repos the ticket touches, their build order, and whether a cross-repo integration test applies.
2KickoffPer repo: planner kicks off (branch + move to In progress + plan); QA planner designs the test plan for the e2e repo.
3BuildThe build role implements in dependency waves — developer (TDD) for code, QA runner (Page-Object) for e2e.
4Open PR/MRThe build role opens the PR right after build, before review, so every reviewer comments on the open PR.
5ReviewOn the open PR: code reviewer (standards + spec) + guardian (security) + performance all review; the dev fixes the combined batch; capped rounds. (Skipped for the e2e repo.)
6MergeEach repo squash-merged upstream → downstream so the web PR shows Merged; each commit recorded.
7IntegrationQA runner runs this ticket's scope against the merged build — the cross-repo join gate (skipped if not needed).
8DistributeEach shippable repo goes to its configured target (e.g. Firebase App Distribution).
9SummaryDocumentor writes the run summary + a per-repo / per-role token & time table.
ℹ️ Safe dry runs
Add --dry-run (e.g. /dev-cycle FM-12 --dry-run) to stop after build & gates — no merge, no integration, no distribution. Great for confirming behaviour without touching the base branch.

05Provider-agnostic by design

The agents never talk to GitHub or Notion directly. They call a thin adapter — a set of shell scripts with a stable command surface — and the adapter dispatches to whichever provider you configured. Swapping GitHub for GitLab, or Notion for Jira, changes a setting, not the agents. Adding a brand-new provider means dropping one file in.

06How to use the aiworks CLI

Three commands cover the full repo lifecycle. Run them from the workspace root — they're all idempotent, so re-running is safe.

add

sync

remove

Registers one repo, clones it, builds the codegraph index, installs the agent skill packs, seeds hooks and settings, and scaffolds a CLAUDE.md + scripts/dev.sh shaped by the repo's own toolchain. Anything already done is skipped.

scripts/aiworks add \
  --url git@github.com:your-org/feeedme-api.git \
  --product backend \
  --lang go \
  --kind generic \
  --distribute none

ℹ️ The workflow CONFIG block is updated for you
When add finishes it regenerates the ── CONFIG ── block at the top of .claude/workflows/dev-cycle.js from workspace.config.yaml (the aiworks config step). Workflows can't read the filesystem at runtime, so that block is their in-source mirror — and it now tracks the config automatically. Nothing to paste.

Reads every repo declared in workspace.config.yaml and runs add for each one. Repos already set up just report SKIP — so this is safe to re-run any time, and ideal for bringing up a fresh clone of the workspace.

# Sync all products
scripts/aiworks sync

# Sync only one product
scripts/aiworks sync feeed-me

# Preview what would run (no changes)
scripts/aiworks sync --dry-run

Workflow: edit workspace.config.yaml first (add a URL under the right product's repos:, with its kind), then run sync — it reads url/kind/lang/distribute/path straight from each entry.

Deregisters a repo: removes its entry from mani.d/<product>.yaml, from workspace.config.yaml, and from the workspace .gitignore. The clone is left on disk unless you add --purge.

# Deregister (keep the clone)
scripts/aiworks remove feeedme-api

# Deregister AND delete the clone
scripts/aiworks remove feeedme-api --purge

# Force-delete even if the tree is dirty
scripts/aiworks remove feeedme-api --purge --force
ℹ️ The dev-cycle CONFIG drops the repo automatically
After deregistering, remove regenerates the REPOS map in .claude/workflows/dev-cycle.js from the updated workspace.config.yaml (the aiworks config step), so the repo disappears from the workflow mirror too — no hand-editing.

07workspace.config.yaml guide

workspace.config.yaml is the single file you fill in to set up an org. Skills, agents, and the dev-cycle workflow all read their slice from it. scripts/aiworks sync reads products: and does every mechanical step for you — clone repos, generate mani.d/, build codegraph indexes, install skill packs, seed hooks, scaffold per-repo CLAUDE.md and scripts/dev.sh.

ℹ️ The workflow CONFIG is regenerated for you
After editing products:, branch_model, tracker, vcs.auto_merge, or planning, run scripts/aiworks config (already run for you by sync / add / remove) — it rewrites the CONFIG block at the top of .claude/workflows/dev-cycle.js from workspace.config.yaml. The workflow script cannot read the filesystem at runtime, so it carries that in-source mirror; aiworks config keeps it in sync, so the dev-cycle always branches/merges against the right targets.

08Why a whole team, not one agent?

A single capable agent can write code. But shipping a multi-repo product needs coordination, specialized review, and a paper trail. Here's how the orchestrated team compares with the alternatives.

Recommended

aiworks — orchestrated team

  • Coordinates many repos in dependency order, automatically
  • Built-in review + security + performance gates
  • Portable across hosts/trackers via adapters
  • End-to-end traceability: ticket → PR → ship
  • One-time config to stand up

Single coding agent

  • Zero setup
  • Fine for a quick, one-repo change
  • No cross-repo coordination
  • Ad-hoc quality & traceability

Manual process

  • Full human judgement
  • No new tooling
  • Slow & people-bound
  • Consistency varies by person/day

09Implementation Plan — stand up aiworks for your org (AI-previewable)

Concrete steps to instantiate this template for a new organization. Each names the real file to touch; the final step is the acceptance check that proves the wiring works without any destructive operation.

  1. Set identity & providers. workspace.config.yaml (copy from workspace.config.example.yaml) — fill org, vcs.provider + vcs.auto_merge, tracker.provider + ticket_prefix + statuses, branch_model, quality_gate, planning (auto_approve / to_html). Then de-brand CLAUDE.md placeholders.
  2. Declare your repos. Add each repo's URL (+ kind) under products[].repos[] in workspace.config.yaml — that's the whole declaration (mani.d/ is generated from it). Or run scripts/aiworks add --url <git-url> --product <name> --kind <flutter-app|appium-e2e|generic> to add one repo imperatively.
  3. Wire the tracker. scripts/tracker/.env (from .env.example) — set TRACKER_PROVIDER and its block; fill the ids/status names in docs/agents/issue-tracker.md.
  4. Wire version control. gh auth login or glab auth login — the provider auto-detects from the origin remote (override via scripts/vcs/.env).
  5. Onboard the repos. scripts/aiworks sync reads products[].repos[] and sets each repo up end-to-end (generates mani.d/, clones, codegraph index, skill packs, hooks, CLAUDE.md + scripts/dev.sh). Idempotent — re-run any time.
  6. Adapt the stack-specific skills. The ── CONFIG ── block in .claude/workflows/dev-cycle.js (REPOS, TICKET_PREFIX, AUTO_MERGE, AUTO_APPROVE_PLAN, PLAN_TO_HTML, STATUS) is generated from the yaml by the previous step — no hand-editing (re-run scripts/aiworks config after any later config change). Then adapt the stack-specific skills (.claude/skills/coding-feature, coding-automate, plan-appium-automate) to your stack.
  7. Acceptance — verify the wiring (no destructive ops): scripts/tracker/get-ticket-details.sh <KEY> prints a ticket · scripts/vcs/default-branch.sh prints the base branch · node --check .claude/workflows/dev-cycle.js passes · then /dev-cycle <KEY> --dry-run runs through build & gates without merging.
